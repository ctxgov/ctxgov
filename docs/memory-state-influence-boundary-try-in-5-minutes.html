Local Memory State Influence Boundary Report

Audit user-supplied local AI memory/context/state files and report which refs are inform-only, blocked, omitted, stale, imported, or unsupported before they are allowed to influence behavior.

Run It

git clone https://github.com/ctxgov/ctxgov.git
cd ctxgov
python3 scripts/run_memory_state_influence_boundary_report.py --input examples/memory-state-influence-boundary/

Then point it at your own local file or directory:

python3 scripts/run_memory_state_influence_boundary_report.py --input ./CLAUDE.md

Bounded directory scans are also supported:

python3 scripts/run_memory_state_influence_boundary_report.py --input ./memory-state/

For CI or product integration, fail the command when blocked refs are found:

python3 scripts/run_memory_state_influence_boundary_report.py --input ./memory-state/ --fail-on-blocked

To emit only the machine-readable gate for a wrapper or CI step:

python3 scripts/run_memory_state_influence_boundary_report.py --input ./memory-state/ --format gate --fail-on-blocked

The blocked-ref gate still writes the JSON, Markdown, and HTML report; it only changes the process exit code for automation.

The JSON report also includes an integration_gate object with default_exit_code, fail_on_blocked_exit_code, blocked/omitted/stale/imported counts, and raw_content_included=false for product integration.

The command writes JSON, Markdown, and HTML reports under .ctxvault/memory-state-governability-overlay/. It performs no provider or model call, no runtime adapter execution, no memory/backend write, no target write, no SARIF upload, and no outreach.

What It Checks

User InputCLAUDE.md, AGENTS.md, MEMORY.md, project notes, MDX context files, JSON/JSONL exports, YAML/YML state files, and TOML state profiles.
Influence BoundaryCandidate refs, inform-only allowed refs, omitted refs, blocked refs, stale or superseded state, and imported context refs.
Action BoundaryAuthority ceiling, final-state assertion, rollback path, deletion propagation, and side-effect boundary.

Output

The input mode writes:

  • influence-boundary-report.json
  • influence-boundary-report.md
  • influence-boundary-report.html

Reports include paths, line refs, structured JSON/TOML/YAML key/value path refs, hashes, signal ids, and decisions. Raw file content is not included. Inputs outside the repository are rendered with input-relative paths such as input/CLAUDE.md, so reports do not default to exposing local absolute paths. Findings include recommendations for the next local review step. Unsupported or over-limit inputs are counted as omitted, with a bounded skipped-input sample retained for review. Malformed JSON/JSONL/TOML state exports are blocked until the export is parseable.

Boundary

This is not a provider integration, runtime adapter, compatibility matrix, support claim, endorsement, security guarantee, benchmark, savings claim, adoption claim, or stable protocol. It is a local influence-boundary audit for reasoning about what proof is missing before memory/context/state becomes operational state. This release is distinct from the prior claim-firewall and Activation X-Ray launches because the user supplies the memory/context/state files.
